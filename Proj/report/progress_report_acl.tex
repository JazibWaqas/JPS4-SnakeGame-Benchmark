\begin{abstract}
We study shortest-path finding on uniform-cost 4-connected grid maps with random obstacles. We compare three classical / modern techniques: Dijkstra's algorithm, A*, and Jump Point Search (JPS4) following recent 4-connected JPS formulations \cite{baum2025}. We report a reproducible preliminary benchmark (mean search time and heap expansions) on synthetic grids, plus a Snake-style visualization for qualitative comparison. Challenges include fair timing across implementations and scaling experiments to larger grids and denser mazes.
\end{abstract}

\section{Problem and motivation}
The task is single-pair shortest paths on a grid graph: cells are 4-connected, each step has unit cost, and blocked cells act as obstacles. This setting models navigation and game pathfinding. We measure \emph{wall-clock search time} and \emph{number of node expansions} (heap pops in our unified search loop) as proxies for work and efficiency.

\section{Algorithms}
\textbf{Dijkstra} explores in increasing distance from the source without goal bias. \textbf{A*} uses a Manhattan heuristic to the goal, reducing expansions while preserving optimality on uniform grids. \textbf{JPS4} prunes symmetric moves on grids and expands fewer nodes before reaching the goal; we use the implementation aligned with 4-connected JPS \cite{baum2025,harabor2011}.

\section{Experimental setup (preliminary)}
We generate \BenchGrid{} grids with independent Bernoulli obstacles at each cell with density $\rho \in \{0.15, 0.28, 0.40\}$. For each trial we sample start and goal on free cells with large separation, require connectivity, and require \emph{all three} methods to return a valid path (otherwise we resample). We run \BenchTrialsPer{} successful trials per density (base seed \BenchSeed{}; run ID \BenchStamp{}). Code path: \texttt{Proj/src/Snake Game Code/pathing\_grid.py} (same as the interactive demo).

\section{Challenges identified}
\textbf{(1)} Synthetic random maps are easy to describe but may not stress the same behaviors as hand-designed mazes. \textbf{(2)} Absolute timings depend on hardware; we report them alongside expansions for fairness. \textbf{(3)} JPS4 vs.\ A* wall-clock can cross over due to implementation overhead at low density---we will document this in the final analysis.
