SLIDE 2 — THE PROBLEM & MOTIVATION
================================================================================

HEADING: Why Does Pathfinding Efficiency Matter?

BULLET POINTS:
  • Grid pathfinding is everywhere: game AI, robot navigation, logistics
  • In a 4-connected grid, each cell links to its 4 cardinal neighbours (up,
    down, left, right) — no diagonals allowed
  • A naive search visits every reachable cell before confirming the shortest path
    → On large maps this is far too slow
  • The question: can smarter algorithms dramatically reduce the number of cells
    they check, without sacrificing the guarantee of finding the shortest path?

WHAT WE'RE COMPARING:
  Three algorithms with different strategies:
    1. Dijkstra   — no sense of direction, checks everything
    2. A*         — uses a heuristic to aim toward the goal
    3. JPS4       — skips over empty space entirely using "jump points"

SPEAKER NOTES:
  Briefly explain the 4-connected constraint and why no diagonals makes this
  harder. Emphasize that all three algorithms guarantee the shortest path —
  what differs is how efficiently they find it. The practical motivation is
  that in real games and robotics, pathfinding runs thousands of times per
  second, so even small improvements matter at scale.

SLIDE 3 — THE ALGORITHMS (METHODOLOGY)
================================================================================

HEADING: Three Algorithms, One Fair Comparison

SECTION: Dijkstra
  - Expands cells in increasing order of distance from the start
  - No direction bias — searches equally in all directions
  - Optimal but slow: has to explore the whole reachable area

SECTION: A*
  - Same as Dijkstra but adds a heuristic: Manhattan distance to the goal
  - This focuses the search toward the goal, skipping areas in the wrong direction
  - On 4-connected grids, Manhattan distance is always a valid lower bound
    → A* still guarantees the shortest path

SECTION: JPS4 (Jump Point Search for 4-connected grids)
  - Asks: "can I skip this cell entirely?"
  - When moving through open space in a straight line, every cell along the way
    has the same cost and opens the same options → no reason to visit each one
  - JPS4 "jumps" through empty space and only stops at JUMP POINTS:
    places where an obstacle forces a real routing decision
  - Introduced by Harabor & Grastien (2011), extended to 4-connected grids
    by Baum (2025) — different pruning rules needed without diagonal moves

KEY DESIGN DECISION:
  All three algorithms share the exact same underlying search loop in our code.
  Only the heuristic and the neighbour-finding function change.
  → Any difference in results is purely algorithmic, not a coding artefact.

VISUAL IDEA:
  Draw a simple grid showing:
  - Dijkstra expanding in a big circle
  - A* expanding in a cone toward the goal
  - JPS4 jumping in straight lines with only a few jump points marked
  (Even a rough hand-drawn style diagram works well for this)

SPEAKER NOTES:
  The key message: JPS4 is smarter about what it ignores. In open space it
  leaps over cells; it only "slows down" when obstacles create forced choices.
  The shared search loop is important to mention — it shows we designed the
  benchmark fairly.

SLIDE 4 — PRELIMINARY RESULTS
================================================================================

HEADING: What the Numbers Show

EXPERIMENTAL SETUP (small box or subtitle):
  • 50×50 grids with random obstacles
  • Three obstacle densities: 15%, 28%, 40% blocked
  • 24 trials per density, fixed random seed (42)
  • Start and goal placed as far apart as possible
  • Only trials where all three algorithms found a valid path were counted

DATA TABLE:
  ┌──────────┬──────────────────┬──────────────────┬──────────────────┐
  │          │   15% blocked    │   28% blocked    │   40% blocked    │
  │          │ Expansions  ms   │ Expansions  ms   │ Expansions  ms   │
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │ Dijkstra │   2062    7.07   │   1659    4.96   │    915    2.73   │
  │ A*       │   1020    3.74   │    629    2.11   │    513    1.70   │
  │ JPS4     │    488    4.43   │    970    6.22   │    682    3.13   │
  └──────────┴──────────────────┴──────────────────┴──────────────────┘

INCLUDE THE FIGURE: results_combined.png
  (two line charts side by side — node expansions and wall-clock time vs density)

KEY FINDINGS (3 bullet points):
  • A* cuts Dijkstra's node count by ~50% at all densities — the heuristic works
  • JPS4 beats both at LOW density (488 vs 1020) — long corridors = big jumps
  • JPS4 WORSE than A* at MEDIUM density (970 vs 629) — obstacles shorten
    corridors, jumps get smaller, overhead stays constant

TIMING NOTE:
  • JPS4 is slower than A* on the clock despite fewer node expansions
  • This is a Python prototype issue — JPS4's inner loop is more complex
  • In C++/Java this overhead disappears (literature confirms this)
  • Node expansions are the fairer comparison at this stage

CORRECTNESS:
  All three algorithms returned equal path lengths on every trial ✓

SPEAKER NOTES:
  Walk through the table row by row. The node expansion story is the main result
  — JPS4 is clearly better on open maps. The timing inversion is something to
  acknowledge honestly; it does not mean JPS4 is worse, it means Python adds
  overhead that a compiled language would not. This is a known issue with
  prototype benchmarks and is discussed in the original paper.

SLIDE 5 — CHALLENGES & NEXT STEPS
================================================================================

HEADING: What We Found Hard — and What's Next

SECTION: Challenges Identified

  1. FAIR TIMING COMPARISON
     Python makes JPS4's inner loop more expensive than it should be.
     Node expansions give a cleaner algorithmic picture right now.
     Plan: profile the inner loop and optimize data access to reduce overhead.

  2. GRID SIZE IS TOO SMALL
     On 50×50, all algorithms finish in under 10ms — noise affects the timing.
     JPS4's advantage compounds with map size (longer corridors = bigger jumps).
     We need 100×100 and 200×200 to see the real performance gap.

  3. DENSITY CROSS-OVER POINT NOT KNOWN
     JPS4 wins at low density, loses at medium density.
     We don't know exactly where the break-even point is.
     Knowing this is practically important: it tells you when to use JPS4 vs A*.

SECTION: Next Plan

SPEAKER NOTES:
