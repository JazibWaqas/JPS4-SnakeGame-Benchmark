\begin{abstract}
We compare three shortest-path algorithms on uniform-cost 4-connected grid maps: Dijkstra's algorithm, A*, and JPS4, a recent adaptation of Jump Point Search for 4-directional movement. All three are guaranteed to return optimal paths. The question is how efficiently they do it. We ran 30 trials per density level across a sweep from 10\% to 40\% obstacle density on both 50$\times$50 and 100$\times$100 grids (420 problem instances per grid size). JPS4 consistently expands fewer nodes than both Dijkstra and A*, reducing expansions by 60--75\% relative to Dijkstra across most density levels. The wall-clock gap is smaller due to JPS4's per-step overhead, and at very low obstacle density the advantage over A* shrinks. All algorithms are implemented in Python, verified against each other for path-length correctness on hundreds of random instances, and wrapped in an interactive Snake demo that lets a viewer watch the three algorithms race on the same board in real time.
\end{abstract}

\section{Introduction}

Shortest-path search on grid maps is one of the most common operations in game AI, robotics, and navigation systems. The choice of algorithm matters a lot in practice: a tile-based game calling a pathfinder every frame cannot afford an algorithm that explores most of the map to reach a nearby goal.

This project targets a specific, well-defined setting: a 4-connected grid where every step costs 1. That constraint rules out 8-directional movement, weighted edges, and any-angle paths. Within those boundaries, we compare three algorithms that all find optimal paths:

\begin{itemize}[nosep]
  \item \textbf{Dijkstra} -- the uninformed baseline. No heuristic, expands outward in distance order.
  \item \textbf{A*} -- adds Manhattan distance as a heuristic to guide search toward the goal.
  \item \textbf{JPS4} -- applies symmetry pruning from Baum (2025) to skip intermediate nodes on straight corridors.
\end{itemize}

The core research question is: does JPS4's pruning give a meaningful practical advantage over A*, and under what conditions?

We built our own implementations of all three algorithms, wrote a test suite that checks path-length agreement across algorithms on hundreds of random instances, and ran a controlled benchmark across two grid sizes and seven density levels. The same code drives an interactive Snake demo where the algorithms can be watched side-by-side.

\section{Background}

\subsection{Dijkstra's Algorithm}

Dijkstra (1959) finds shortest paths in graphs with non-negative edge costs by expanding nodes in order of their distance from the source~\cite{dijkstra1959}. On a uniform-cost grid, this is equivalent to a wavefront that expands outward in concentric rings. It is complete and optimal but has no goal-directed bias, so it tends to explore a large fraction of the reachable region before terminating.

\subsection{A* Search}

Hart, Nilsson, and Raphael (1968) augment Dijkstra with a heuristic estimate $h(n)$ of the remaining distance to the goal~\cite{hart1968}. The node priority becomes $f(n) = g(n) + h(n)$, where $g(n)$ is the true cost from the start. With an admissible heuristic (one that never overestimates), A* is still optimal and typically expands far fewer nodes than Dijkstra by focusing the search toward the goal.

On a 4-connected grid with unit costs, Manhattan distance is the standard admissible heuristic:
\[
  h(n) = |x_n - x_{\text{goal}}| + |y_n - y_{\text{goal}}|
\]

\subsection{Jump Point Search and JPS4}

Jump Point Search (JPS), introduced by Harabor and Grastien (2011) for 8-connected grids, addresses a different source of inefficiency~\cite{harabor2011}. Even with a good heuristic, A* still expands many \emph{symmetric} paths: multiple routes that have the same cost but differ only in the order of moves along a straight corridor. JPS prunes these by identifying \emph{jump points}, the only nodes at which a symmetric path can be distinguished from another.

Baum (2025) adapted this idea to 4-connected grids, producing JPS4~\cite{baum2025}. The key differences from classical JPS are a horizontal-first canonical ordering and revised pruning rules that account for the absence of diagonal movement. Horizontal moves always return all natural neighbors; vertical moves add forced neighbors only when an obstacle beside the parent would otherwise hide a shorter path around it. Jump points are then placed wherever an obstacle creates such a forced neighbor along the scan direction.

The result is that JPS4 can traverse long straight corridors in a single ``jump'' rather than expanding every cell along them, at the cost of more work per step.

\section{Implementation}

\subsection{Core Algorithms}

All three algorithms are implemented in \texttt{pathing\_grid.py} around a shared A*-style priority queue loop (in \texttt{helper.py}). This design lets the three modes share the same correctness infrastructure and differ only in how they generate successors:

\begin{itemize}[nosep]
  \item \textbf{Dijkstra} uses the same 4-neighbor successor function as A* but passes a zero heuristic, so nodes are popped in pure distance order.
  \item \textbf{A*} uses Manhattan distance as the heuristic and expands all 4 passable neighbors at each step.
  \item \textbf{JPS4} replaces the neighbor expansion with a two-phase operation: (1) prune neighbors based on the incoming direction and adjacent obstacles, (2) jump along each pruned neighbor's direction until a jump point, obstacle, or the goal is found.
\end{itemize}

The JPS4 successor function returns jump points directly, potentially skipping over large swaths of the grid in a single step. The path reconstruction then fills in the intermediate cells.

\subsection{Grid and Connectivity}

The grid is a 2D Boolean array where \texttt{True} means blocked. The \texttt{PathingGrid} class maintains a Union--Find structure over free cells so that reachability queries (start/goal in the same connected component) run in near-constant time without a full BFS.

\subsection{Snake Demo}

The same \texttt{PathingGrid} drives an interactive Snake game built in Tkinter. The game runs three consecutive rounds on the same frozen board, one per algorithm, and displays each algorithm's search time, expansion count, and path length in a HUD strip below the grid. This lets a viewer see the behavioral difference between algorithms at the same density without confounding variables.

When the current food cell becomes temporarily unreachable (blocked by the snake's own body), the game falls back to a tail-following recovery strategy and relocates food to a reachable cell, keeping the demo stable without masking the pathfinder comparison.

\subsection{Test Suite}

The test suite in \texttt{tests/} covers four properties:

\begin{enumerate}[nosep]
  \item \textbf{Path length agreement:} Dijkstra, A*, and JPS4 return the same path length on every solvable instance sampled.
  \item \textbf{Expansion ordering:} Dijkstra never expands fewer nodes than A* on the same instance (sanity check on the heuristic benefit).
  \item \textbf{Reachable food selection:} the snake's food placement always picks a cell reachable from the snake's current head.
  \item \textbf{Recovery correctness:} the previously reproducible unreachable-food loop no longer occurs.
\end{enumerate}

All tests pass on the current codebase.

\section{Experimental Setup}

\subsection{Grid Generation}

Grids are synthetic: for each trial, every cell is independently set as blocked with probability $\rho$ (the obstacle density). We use two grid sizes: 50$\times$50 and 100$\times$100.

Start and goal cells are placed to maximize path length. We first pick a random free cell, BFS to find the farthest reachable cell from it (call it $A$), then BFS again from $A$ to find the farthest reachable cell from $A$ (call it $B$). The trial uses $A$ as start and $B$ as goal. This ``double BFS'' trick avoids trivially short queries and produces instances that stress the algorithms more uniformly.

Each trial is run with all three algorithms on the same board. This controls for instance difficulty and lets us compare algorithms on identical queries.

\subsection{Parameters}

\begin{itemize}[nosep]
  \item \textbf{Grid sizes:} 50$\times$50 and 100$\times$100.
  \item \textbf{Density levels:} 0.10, 0.15, 0.20, 0.25, 0.30, 0.35, 0.40.
  \item \textbf{Trials per condition:} 30.
  \item \textbf{Seed:} fixed per density level (seed $= 99 + 7777 \times i + W$), ensuring reproducibility.
  \item \textbf{Metric -- expansions:} the number of nodes popped from the priority queue (heap pops). This is the primary algorithmic efficiency metric, hardware-independent.
  \item \textbf{Metric -- time:} Python \texttt{time.perf\_counter} wall-clock time in milliseconds. These numbers are hardware-specific and noisier than expansion counts.
\end{itemize}

Instances where any algorithm fails to find a path are excluded from that trial (none occurred in this study; the Union--Find pre-filter ensures we only run solvable instances).

JPS4 consistently expands the fewest nodes at every density level on both grid sizes. On the 100$\times$100 grid at $\rho = 0.10$, JPS4 expands 2,379 nodes versus 8,988 for Dijkstra and 7,903 for A*, a reduction of 73\% relative to Dijkstra and 70\% relative to A*. This large gap at low density makes sense: open maps contain many long straight corridors where JPS4 skips entire rows or columns in a single jump, while A* must still expand every cell.

At higher densities the gap narrows. With many obstacles, paths become more tortuous and corridors are shorter, so there is less symmetry for JPS4 to exploit. At $\rho = 0.40$ on the 100$\times$100 grid, JPS4 still reduces expansions by 51\% over Dijkstra and 39\% over A*, but the advantage is smaller than at low density.

\subsection{Wall-Clock Time}

Wall-clock time tracks the expansion reduction but with more noise. On the 50$\times$50 grid, Dijkstra runs in roughly 6--8 ms at low density, A* in 4--6 ms, and JPS4 in 3--4 ms. The time gap is proportionally smaller than the expansion gap because JPS4's jump logic is more expensive per iteration than A*'s simple 4-neighbor lookup.

At $\rho = 0.35$ on the 50$\times$50 grid, JPS4 and A* produce nearly identical runtimes (2.75 ms vs 2.70 ms) even though JPS4 expands 40\% fewer nodes. This illustrates the overhead cost: each JPS4 step scans along a direction until it hits an obstacle or a forced neighbor, which is not free.

On the 100$\times$100 grid the time advantage for JPS4 recovers. At $\rho = 0.10$, JPS4 takes 18.9 ms versus 30.8 ms for A* and 32.4 ms for Dijkstra. The reason is that on a larger grid, skipping a long corridor saves proportionally more total work, making the overhead cost worthwhile.

\subsection{Path Correctness}

Every trial confirmed that all three algorithms return the same path length. This is expected (all three are optimal on uniform-cost grids), but it serves as an ongoing regression check. The test suite formalizes this: 500 random instances are checked each run, and the suite has found no path-length disagreements since the JPS4 implementation was finalized.

\section{Discussion}

\subsection{When JPS4 Helps Most}

The data consistently shows JPS4's advantage is largest on:
\begin{itemize}[nosep]
  \item \textbf{Larger grids.} The 100$\times$100 grid sees bigger relative gains than 50$\times$50 at every density. Longer corridors mean more nodes per jump.
  \item \textbf{Lower obstacle density.} Open maps have more straight-line symmetry, so more intermediate nodes can be pruned. At $\rho = 0.10$ on 100$\times$100, JPS4 expands 3.4$\times$ fewer nodes than Dijkstra.
\end{itemize}

\subsection{When JPS4 Helps Less}

At high density ($\rho \geq 0.35$) the paths become twisty and jump opportunities shrink. JPS4 still wins on expansion count, but the per-step overhead narrows the wall-clock gap noticeably on smaller grids. In such conditions, standard A* is a simpler and nearly as fast choice.

This matches the observation in Baum (2025): JPS4 is not a universal replacement for A*; it is a specialization for maps with enough open space or repeated symmetry.

\subsection{Dijkstra as a Baseline}

Dijkstra's role here is mostly pedagogical. It confirms that the heuristic in A* actually helps (A* consistently beats Dijkstra on expansions), and it gives both A* and JPS4 a common reference point. In a real system you would not use Dijkstra for single-query grid pathfinding, but it is the clearest illustration of what the heuristic and the pruning are each contributing.

\subsection{Limitations}

\begin{itemize}[nosep]
  \item \textbf{Implementation language.} Everything runs in interpreted Python. Absolute times are not representative of C or C++ performance. Ratios and expansion counts are more meaningful.
  \item \textbf{Synthetic grids.} Random independent-cell obstacle placement produces a specific texture. Real maps (mazes, rooms, corridors) would likely show different density-performance relationships.
  \item \textbf{Single heuristic weight.} We used $w = 1$ for A* (exact admissible heuristic). Weighted A* ($w > 1$) trades optimality for speed and is not evaluated here.
  \item \textbf{No preprocessing.} JPS4 is an online method. Algorithms that allow offline preprocessing (such as HPA* or Contraction Hierarchies) are not compared.
\end{itemize}

\section{Conclusion}

We implemented and benchmarked Dijkstra, A*, and JPS4 on uniform-cost 4-connected grid maps across 420 problem instances per grid size. The results confirm that JPS4 reduces node expansions substantially relative to both Dijkstra and A* across all tested densities, with the largest gains on larger, more open maps. The reduction in wall-clock time is real but smaller, because JPS4's jump logic costs more per step than a plain 4-neighbor expansion.

For a practical recommendation: on 4-connected uniform-cost grids where maps are reasonably open and path queries are frequent, JPS4 is the right choice. On dense or heavily cluttered maps, or when simplicity matters, A* with Manhattan distance is hard to beat. Dijkstra belongs in the textbook.

The Snake demo makes the tradeoff concrete and visible: on a sparse board you can watch JPS4 finish its search while A* is still expanding nodes, and on a crowded board the three algorithms converge to nearly the same behavior.
